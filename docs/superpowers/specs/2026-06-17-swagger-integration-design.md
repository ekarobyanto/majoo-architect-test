# Design Spec: Swagger/OpenAPI Integration

**Status:** Draft
**Date:** 2026-06-17
**Topic:** Swagger documentation for Go Fiber API

## 1. Problem Statement
The project lacks interactive API documentation. Developers need a way to explore, test, and understand endpoints without reading through handler code manually.

## 2. Proposed Solution
Implement automated Swagger documentation using `swag` (swaggo) and host it directly within the Fiber application using `gofiber/swagger`.

## 3. Architecture & Tools
- **Swagger Generator**: `swag` (v1.16+)
- **Fiber Middleware**: `github.com/gofiber/swagger`
- **Output Directory**: `docs/` (generated by swag)

## 4. Implementation Details

### 4.1. General API Metadata
General annotations will be placed in `cmd/api/main.go`:
- `@title`: Blog System API
- `@version`: 1.0
- `@description`: This is a sample blog system server.
- `@termsOfService`: http://swagger.io/terms/
- `@contact.name`: API Support
- `@contact.email`: support@example.com
- `@license.name`: Apache 2.0
- `@host`: localhost:3000
- `@BasePath`: /
- `@securityDefinitions.apikey`: ApiKeyAuth (for JWT)

### 4.2. Handler Annotations
Each handler (e.g., Health, Auth, Posts) will be annotated with:
- `@Summary`
- `@Description`
- `@Tags`
- `@Accept`
- `@Produce`
- `@Param` (for inputs)
- `@Success` / `@Failure` (for outputs)
- `@Router` (path and method)
- `@Security` (where applicable)

### 4.3. Registration
- Import the generated `docs` package in `internal/platform/server/server.go`.
- Register the route: `app.Get("/swagger/*", swagger.HandlerDefault)`.

### 4.4. Automation
Add a target to the `Makefile`:
```makefile
swagger:
	swag init -g cmd/api/main.go -o ./docs --parseDependency --parseInternal
```

## 5. Success Criteria
- Running `make swagger` generates/updates the `docs/` folder.
- Accessing `http://localhost:3000/swagger/index.html` displays the interactive Swagger UI.
- All existing endpoints (Health check) are documented and testable from the UI.

## 6. Self-Review
- **Placeholder scan**: None.
- **Internal consistency**: Architecture matches Fiber/sqlx patterns.
- **Scope check**: Focused solely on Swagger setup.
- **Ambiguity check**: BasePath and Host are clearly defined for local dev.
